# SQL 脚本部署说明

本文档用于指导项目**首次部署**和**版本升级**时 SQL 脚本的执行顺序、路径及注意事项，确保数据库结构和数据正确初始化 / 更新。

## 一、脚本目录结构

项目根目录下`SQL`文件夹为数据库脚本统一存放目录，层级结构如下：

```plaintext
项目根目录/
└── SQL/
    ├── all/       # 全量初始化脚本（首次部署专用）
    │   ├── DDL/   # 数据定义语言脚本（建库、建表、建索引、建约束等）
    │   └── DML/   # 数据操作语言脚本（初始化基础数据、字典数据等）
    └── add/       # 增量更新脚本（首次部署+升级均使用）
        ├── DDL/   # 增量结构变更脚本（新增表、修改字段、新增索引等）
        └── DML/   # 增量数据变更脚本（新增基础数据、更新业务数据等）
```

## 二、脚本执行规则

### 1. 核心执行顺序（严格遵守，不可颠倒）

- 首次部署（全新环境，无数据库基础）：

  `all/DDL`  → `all/DML`  →  `add/DDL`  →  `add/DML`

- 版本升级（已有旧版本数据库，仅更新变更）：

   `add/DDL` → `add/DML`

### 2. 子目录执行规则

同一层级文件夹下的脚本，**按文件名称升序依次执行**（add目录，建议脚本命名规范：`20260101_01_dml/ddl_修复错误.sql`）。

## 三、分场景执行步骤

### 场景 1：项目首次全新部署

1. 准备工作：确认数据库服务已启动，已创建项目对应空数据库（若`all/DDL`包含建库脚本，可跳过手动建库）；
2. 执行`SQL/all/DDL`目录下所有脚本，完成数据库表、索引、约束等结构初始化；
3. 执行`SQL/all/DML`目录下所有脚本，完成项目基础数据、字典数据初始化；
4. 执行`SQL/add/DDL`目录下所有脚本，完成首次部署的增量结构更新；
5. 执行`SQL/add/DML`目录下所有脚本，完成首次部署的增量数据更新；
6. 验证：检查数据库结构和数据是否完整，无报错则部署完成。

### 场景 2：项目版本升级

1. 准备工作：备份当前数据库（防止升级失败数据丢失）；
2. 仅执行`SQL/add/DDL`目录下所有脚本，完成数据库结构增量变更；
3. 仅执行`SQL/add/DML`目录下所有脚本，完成数据增量变更；
4. 验证：检查升级后功能正常，无报错则升级完成。

## 四、注意事项

1. **严格按顺序执行**：DDL 脚本（结构）必须优先于 DML 脚本（数据）执行，否则会因表不存在导致执行失败；
2. **禁止修改脚本**：请勿随意修改`SQL`文件夹下的脚本内容、名称和路径，避免执行异常；
3. **升级不执行 all 脚本**：升级场景**绝对不要执行`all/`目录下的任何脚本**，否则会覆盖 / 删除原有数据；
4. **异常处理**：执行过程中若出现报错，立即停止执行，排查报错原因（如权限不足、脚本语法错误、数据冲突），修复后再继续；
5. **环境适配**：脚本默认适配项目指定数据库类型（**MySQL/MariaDB**），跨数据库执行需提前适配语法。
